# Plan: freechains begs — send/recv (official + transparent)

## Context

A **beg** is a post awaiting promotion (signed or unsigned
contribution that doesn't yet have a like to grant it
reputation weight).

Two distinct flows:

- **Official begs**: `refs/begs/*` ref namespace, explicit
  protocol, pushed/pulled via freechains send/recv.
- **Transparent begs**: plain `git push` from peers that
  don't speak freechains; content arrives on `main` as
  begs without the protocol dance.

## Prior work

`sync.md` steps 6-8 (deferred): beg registration, cross-host
like, beg pruning. Restart from those ideas with the new

## Official begs

### Push flow (sender)

`freechains chain <n> sync send <remote>` additionally:

```
git push -o freechains=true -o url=<A> <remote> \
    main refs/begs/*:refs/begs/*
```

Pushes both `main` and all local `refs/begs/*` in one push.

### Hook (receiver)

`pre-receive` must accept either:

- exactly one ref == `refs/heads/main`, or
- one ref == `refs/heads/main` plus `refs/begs/*` refs

Reject everything else. For each `refs/begs/*`:

1. Read the beg commit contents.
2. Register in `.freechains/state/posts.lua` as `state=beg`.
3. Register in `.freechains/state/authors.lua` if new key.

Invokes (or replaces) recv inside hook to do the work.

### Pruning

After a beg is merged into main (via a like that
legitimizes it), delete `refs/begs/<id>`. Sender's next
send drops the ref from remote too (`git push :refs/begs/<id>`).

### Lifecycle

```
1. author A creates a beg   -> writes commit to refs/begs/<id>
2. A sends to B             -> refs/begs/<id> appears on B
3. B likes the beg          -> commit on main that consumes ref
4. next sync: beg ref pruned on both sides
```

## Transparent begs

### Trigger

`pre-receive` receives a push **without** `freechains=true`
option. Treat as transparent.

### Rules

- FF only. Non-FF rejected (no merge protocol).
- Every pushed commit on `main` is registered as a **beg**
  regardless of signature status.
    - signed commit -> beg with author key recorded
    - unsigned commit -> beg without author
    - forged signature -> beg without author (don't reject)
- No state commit overwrite — the sender's state (if any)
  is ignored; receiver computes its own.
- Hook exit 1 at the end, like official send.

### Hook code path

Probably a new subcommand `freechains ... sync _beg <tip>`
that:

1. `git fetch`-free, data already local via the push.
2. Walk `old..new`, for each commit treat as beg.
3. Append to posts.lua/authors.lua.
4. Write state, new state commit.
5. Update refs/heads/main to new state commit hash.

## Key design changes during implementation

- **`--no-ff` for like-merges**: beg promotions always create 2-parent commits (merge with `Freechains: like` trailer). Simplifies replay detection.
- **Climb beg flag (Option H)**: `climb`/`meet`/`commit` carry a `beg` arg. On merge with `like` trailer, right side marked beg → all visited posts get `T.beg=true` in apply.
- **State `'beg'` vs `'blocked'`**: renamed to `'beg'` for clarity.
- **Like auto-detects beg target**: `target_is_beg = G.posts[id].state == "beg"` in recv's `commit()` — promotes without caller passing `T.beg`.
- **`apply('like')` sets author time on promotion**: avoids consolidation crash on nil time.
- **`apply('post')` strict for `T.beg=false`**: 0-rep signed → still rejected (no implicit beg).
- **`like.lua` appends beg-branch commits to G.order**: matches recv's traversal so `order.lua` agrees on both sides.
- **Recv fetch refspec**: `main refs/begs/*:refs/begs/* --prune`.
- **Recv merge uses `--ff-only <hash>`**: avoids multi-refspec FETCH_HEAD ambiguity.
- **Hook accepts `refs/heads/main` + `refs/begs/*`**: rejects others with `unexpected ref : '<ref>'`.

## Open

- Who signs the beg commits' state commit on the receiver
  (transparent path)? Needs a receiver-owned key, or use
  an unsigned state commit for beg-only updates.
- Naming of refs/begs/<id> — hash? sequential? timestamp?
- Can official and transparent pushes coexist in the same
  chain, or do peers choose one mode per relationship?
- Concurrent pushes while beg refs are being pruned —
  lockless protocol safety.
