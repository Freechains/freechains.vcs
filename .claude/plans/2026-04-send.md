# Plan: freechains chain sync send

## Context

- recv was done; send was stub.
- Final design:
    - sender pushes `main` with push options
      (`freechains=true`, `url=<sender's repo>`).
    - receiver's `pre-receive` hook invokes
      `freechains sync recv <url>` which does the real
      fetch + validate + merge on receiver.
    - hook always exits 1 (push itself rejected) so git's
      own ref update doesn't conflict with the ref update
      recv performed during the hook.
    - sender recognizes `Freechains: OK` in stderr as
      success.
- Assumption: working tree always clean
  (no uncommitted posts).

## Approach: push-option + hook runs recv

```
A (sender):
    git push -o freechains=true -o url=<A-path> B main

B (receiver) pre-receive hook:
    1. assert single ref == refs/heads/main
    2. read push options (freechains, url)
    3. unset git env leakage (GIT_DIR, GIT_QUARANTINE_PATH,
       GIT_OBJECT_DIRECTORY, ...)
    4. freechains --root=... chain <n> sync recv <url>
        (recv fetches from sender, validates, merges,
         updates local refs + WT)
    5. print "Freechains: OK" to stderr
    6. exit 1 (reject the original push)

A reads stderr:
    - "Freechains: OK" -> treat as success
    - otherwise -> propagate error
```

Consequences:

- Validation **reused** from existing `recv` — no
  duplicate Phase 1 code in the hook.
- All `recv` side effects (merge commit, state commit,
  WT refresh) happen naturally.
- The "push rejected" git message is benign — data is in.

## Future

### Accept transparent begs from standard `git push`

Current hook rejects pushes lacking `freechains=true`.

Future mode: when the option is absent, accept the push
as a source of **beg commits** (unsigned / unknown
authors).

Rules:
- FF only (no merge logic available without sender URL)
- every pushed commit treated as a beg regardless of
  signature status
- hook runs Phase 1 replay but with relaxed rules:
  missing/forged signatures -> beg instead of reject
- non-FF -> reject

Lets plain `git push remote main` work for peers that
don't speak freechains -- their contributions land as
begs awaiting likes for legitimization.
